# Adaptive Agent Orchestrator v0.4.2-beta.1

[简体中文](#简体中文) · [English](#english)

## 简体中文

### 这是什么

Adaptive Agent Orchestrator 是一个面向 Codex 的上下文高效型多 Agent
协调 Skill。它的目标不是尽可能多地创建 Worker，而是在任务确实存在独立
工作流时，减少重复读取、重复推理和无意义的协调；小型、强顺序和高上下文
重叠任务仍由主 Agent 直接完成。

这一版把“角色设计”和“是否创建 Worker”彻底分开，并为论文共同撰写、行业
角色选择和 Worker 创建透明度建立了可执行规则。

### 为什么值得升级

#### 1. 角色不再等于 Worker

角色是责任、边界和验收标准；Worker 是一次真实执行。选中或定义角色不会
自动创建子 Agent。只有该角色拥有独立产物、独立上下文或确实可并行的工作
时，主 Agent 才会把它实例化为 Worker。

创建前，用户会看到：

- 角色负责什么；
- 为什么当前任务需要它；
- 它会读取和排除哪些上下文；
- 它将交付什么；
- 它是临时角色、项目角色还是用户长期拥有的角色。

创建后，系统只报告实际创建的 Worker、真实身份和当前状态，不用计划中的
角色冒充已经运行的 Agent。

#### 2. 最多四个 Worker，但不会为了凑数而创建四个

四个是上限，不是默认值。主 Agent 会同时计算直接 Worker 与持久运行中的
Worker；一次持久运行内的确定性脚本也会执行相同上限。一个任务可能使用
零个、一个或最多四个 Worker，取决于独立工作流是否真实存在。

#### 3. 四套紧凑行业角色包

本版加入按需加载的角色目录，每个行业最多暴露三至四个互补角色：

- 供应链：需求与库存、供应与采购、网络与履约、风险与控制；
- 软件开发：架构与边界、实现、测试与可靠性、安全与审阅；
- 美术与创意制作：创意方向、视觉执行、生产适配、质量审查；
- 股票研究：公司与行业、财务与估值、催化剂与风险、独立投资审阅。

这些角色是候选职责，不是固定编制。Skill 只加载被选中的角色合同，不把
完整行业手册复制进每个 Worker prompt。涉及实时行情、财报或投资判断时，
仍需使用最新的一手资料；角色包不替代数据核验。

#### 4. 论文项目从“主 Agent 独写”升级为受控共同撰写

新增 `manuscript_profile`，区分三种职责：

- 共同作者：拥有明确章节和对应证据；
- 研究贡献者：交付分析、实验或材料，不直接决定全文结构；
- 独立审稿人：只在质量门审阅，不参与起草被审内容。

主 Agent 继续负责论证主线、跨章节一致性、最终文风和发布决策，但不再
天然垄断全部正文。这样可以让方法专家和行业专家真正贡献内容，同时避免
多人同时改写同一段造成冲突。

#### 5. 直接 Worker 快速路径保持轻量

单个临时、只读、边界明确的 Worker 仍可绕过持久计划、日志和角色存储。
如果用户明确要求长期角色、后续复用或可恢复执行，则自动转入持久路径，
不会因快速路径而丢失角色身份。

#### 6. 更严格的启动、授权和重试边界

- 尚未真正创建的 Worker 不占重试和 Worker 槽位；
- 自动授权必须引用已经存在的项目相对路径；
- Worker 预览明确显示排除上下文、会话方式和协作拓扑；
- 差量重试必须绑定同一节点在真实失败运行中的证据，不能伪造失败后跳过
  完整执行。

### 如何使用

安装到 Codex Skill 目录后，直接描述目标即可：

```text
用 adaptive-agent-orchestrator 完成这篇预测模型论文。
先判断是否真的需要 Worker；创建前说明每个角色、必要性、输入和交付物。
方法章节可由方法共同作者负责，供应链章节可由领域共同作者负责，
最后由主 Agent 统一论证和文风。
```

行业任务示例：

```text
分析这家公司的季度业绩。最多启用四个股票研究角色，
但只创建有独立工作流的 Worker，并在创建前解释角色定义。
```

如果任务很小，预期行为是保持单 Agent；没有必要显式配置 Token budget、
DAG 或角色数量。

### 安装与升级

把仓库中的 `skills/adaptive-agent-orchestrator` 复制到：

```text
%USERPROFILE%\.codex\skills\adaptive-agent-orchestrator
```

然后重新启动 Codex 或开启一个新任务，使 Skill 目录被重新发现。

### 验证结果

- 369 项自测断言通过；
- 36 个负向用例通过；
- 15 个 PowerShell 脚本通过语法解析；
- 快速结构校验通过；
- 本地安装副本与发布副本哈希一致；
- 独立 Worker 审阅结论为绿色，无 P0/P1 问题。

### 当前限制

- 四 Worker 的跨路径上限由主 Agent 控制器负责；确定性脚本只能在单个持久
  运行内部强制执行，当前没有跨运行的机器级共享账本。
- 角色包提供职责边界，不提供领域事实；股票、供应链和软件安全等高时效或
  高风险任务仍需要最新数据和独立核验。
- 本版以减少上下文复制和无效协调为设计目标，但不宣称能预先准确预测开放
  式任务的总 Token，也不在缺少公平实测时承诺固定节省比例。

## English

### What it is

Adaptive Agent Orchestrator is a context-efficient coordination Skill for
Codex. It does not try to create as many Workers as possible. It coordinates
only genuinely independent workstreams while reducing repeated reads,
duplicated reasoning, and unnecessary governance. Small, sequential, and
high-overlap tasks stay with the main agent.

This release separates role design from Worker creation and adds enforceable
rules for manuscript co-authorship, industry role selection, and transparent
Worker activation.

### Why upgrade

#### 1. A role is no longer a Worker

A role defines responsibility, boundaries, and acceptance criteria. A Worker
is a real execution instance. Selecting or defining a role does not create an
agent automatically. A role is materialized only when it owns an independent
artifact, disjoint context, or useful parallel work.

Before creation, the user sees the role's purpose, necessity, included and
excluded context, deliverable, and lifetime. After creation, the system reports
only the Workers that actually exist, with their real identity and state.

#### 2. Four Workers is a ceiling, not a target

The main agent counts direct and durable Workers together, while deterministic
scripts enforce the same ceiling inside a durable run. A task can use zero,
one, or at most four Workers. The Skill never creates four merely because four
roles are available.

#### 3. Four compact industry role packs

On-demand catalogs provide three or four complementary responsibilities for:

- supply chain: demand and inventory, sourcing, fulfillment, and risk;
- software development: architecture, implementation, reliability, and
  security review;
- creative production: direction, visual execution, production adaptation,
  and quality review;
- public equity research: company and industry, financials and valuation,
  catalysts and risk, and independent investment review.

These are candidate responsibilities, not a fixed team. Only the selected
contract is loaded into context. Time-sensitive market, filing, and investment
claims still require current primary sources.

#### 4. Manuscripts gain bounded co-authorship

The new `manuscript_profile` distinguishes:

- co-authors who own bounded sections and their evidence;
- research contributors who supply analysis or experiments;
- independent reviewers who do not draft the work they judge.

The main agent retains the argument spine, cross-section consistency, final
voice, and publication decision without monopolizing every paragraph.

#### 5. The direct-Worker path stays lightweight

One temporary, read-only, bounded Worker can still bypass durable plans,
journals, and stored roles. A request for persistent identity, later reuse, or
recoverable execution switches to the durable path instead of silently losing
the role.

#### 6. Tighter startup, authorization, and retry rules

- an unmaterialized Worker consumes neither a retry nor a Worker slot;
- automatic authorization requires an existing project-relative path;
- previews show excluded context, session policy, and topology;
- delta retry must reference real failure evidence for the same node and run.

### How to use it

After installation, describe the outcome rather than configuring a workflow:

```text
Use adaptive-agent-orchestrator for this forecasting manuscript.
Decide whether Workers are actually necessary. Before creating each Worker,
explain its role, necessity, inputs, exclusions, and deliverable.
Allow a methods co-author and a supply-chain co-author to own bounded sections,
then let the main agent unify the argument and voice.
```

For a small task, the expected behavior is to remain single-agent. Users do not
need to configure a Token budget, DAG, or desired role count.

### Installation and upgrade

Copy `skills/adaptive-agent-orchestrator` into:

```text
%USERPROFILE%\.codex\skills\adaptive-agent-orchestrator
```

Restart Codex or open a new task so the Skill directory is rediscovered.

### Validation

- 369 self-test assertions passed;
- 36 negative cases passed;
- all 15 PowerShell scripts parsed successfully;
- quick structural validation passed;
- the installed and release copies have matching hashes;
- independent Worker review returned green with no P0/P1 findings.

### Current limitations

- The main-agent controller enforces the four-Worker ceiling across direct and
  durable paths. Deterministic scripts can enforce it only inside one durable
  run; there is no machine-wide cross-run ledger yet.
- Role packs define responsibilities, not domain facts. Current data and
  independent verification remain necessary for time-sensitive or high-risk
  work.
- The design targets lower duplicated context and coordination overhead. It
  does not claim to predict the total Token use of open-ended work or promise a
  fixed saving percentage without fair end-to-end evidence.
